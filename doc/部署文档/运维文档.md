# SafeMask AI 模型下载鉴权代理 — 运维文档

## 1. 架构回顾

```
客户端 → https://www.xiao.org.cn/health      → Nginx → server.js :35347
客户端 → https://www.xiao.org.cn/download?t=… → Nginx → server.js :35347
                                                    ↓ 302
客户端 → https://obs.be.com:9004/… (直连，不经过鉴权服务器)
```

**数据流原则**：鉴权服务器只验证令牌和返回 302，文件数据从客户端直连 OSS 下载，鉴权服务器带宽成本为零。

**端口一览**：

| 端口 | 进程 | 监听地址 | 协议 |
|------|------|----------|------|
| 35347 | server.js | 0.0.0.0 | HTTP |
| 443 | nginx | 0.0.0.0 | HTTPS |
| 80 | nginx | 0.0.0.0 | HTTP → 301 跳转 HTTPS |

## 3. 健康检查机制

### 3.1 端点说明

`GET /health` 返回 `{"status":"UP","timestamp":<毫秒>}`。无 CPU 消耗，无数据库查询，仅有 JSON 序列化开销。

### 3.2 前端自动降级流程

```
用户点击下载
  │
  ├─ 前端提取 URL base（如 https://www.xiao.org.cn）
  ├─ fetch GET /health（1.5 秒超时）
  │    ├─ 200 OK → 调用 Rust start_model_download（正常流程）
  │    └─ 失败 → 跳过当前 URL，立即尝试下一个
  │
  └─ model_download_urls 列表
       ├─ [0] https://www.xiao.org.cn/download?token=...（鉴权代理）
       ├─ [1] https://obs.be.com:9004/...（OSS 直连，fallback）
       ├─ [2] https://r2.cloudflarestorage.com/...（R2 fallback）
       └─ [3] https://github.com/...（GitHub fallback）
```

> 健康检查的目的是避免用户在鉴权服务宕机时等待 300 秒的 Rust 下载超时才触发 Fallback。1.5 秒预检失败，立即跳转下一个 URL。

## 4. 部署验证

### 4.1 verify.sh 命令

```bash
# 6 项基础验证（直连 server.js）
bash /opt/safemask-auth/verify.sh

# 通过 Nginx 域名验证
bash /opt/safemask-auth/verify.sh --url https://www.xiao.org.cn

# 完整验证 + OSS 文件可达性检查
bash /opt/safemask-auth/verify.sh --url https://www.xiao.org.cn --download-check

# 指定密钥文件路径
bash /opt/safemask-auth/verify.sh --key-file /opt/safemask-auth/config/hmac.key
```

### 4.2 验证项详解

| # | 测试内容 | 期望 HTTP | 故障定位 |
|---|----------|-----------|----------|
| 1 | `GET /health` | 200 | 服务是否存活 |
| 2 | `GET /download`（无 token） | 400 | 参数校验是否生效 |
| 3 | `GET /`（错误路径） | 404 | 路由是否严格 |
| 4 | `GET /download?token=invalid` | 403 | HMAC 签名验证 |
| 5 | `GET /download?token=<有效>` | 302 | 完整鉴权 + 重定向，确认 Location 头指向 OSS |
| 6 | 同一 token 再请求 | 429 | 频率限制，确认计数生效 |
| 7 | `--download-check` 跟随 302 | 200 + bytes | OSS 直连链路可达 |

### 4.3 verify.sh 退出码

- `0`：所有测试通过
- `1`：至少一项测试失败

## 5. 密钥轮换

触发条件：密钥泄露、定期安全审计、团队成员离岗。

```bash
# 第 1 步：服务器生成新密钥
sudo bash /opt/safemask-auth/init-key.sh --force

# 第 2 步：重启鉴权服务（自动加载新密钥）
sudo bash /opt/safemask-auth/deploy.sh --restart-only

# 第 3 步：读取新密钥
cat /opt/safemask-auth/config/hmac.key

# 第 4 步：本地用新密钥重新构建 Rust 二进制
$env:SAFEMASK_HMAC_KEY = "上一步的密钥"
cargo build -p SafeMask --release

# 第 5 步：分发新二进制给所有用户
```

> 旧令牌在 `TOKEN_TTL_SECS`（1 小时）后自动过期，轮换过程中至多 1 小时的不一致窗口，无需停机。

## 7. 故障排查

### 7.1 502 Bad Gateway

**原因**：Nginx 连接不到后端 server.js。

```
curl -i http://127.0.0.1:35347/health
    ↓
  000 / 拒绝连接    200 OK
    ↓                ↓
server.js 未启动    Nginx proxy_pass 端口错
    ↓                检查 nginx.conf 中的端口
pm2 logs safemask-dl-auth
```

**诊断流程**：

```bash
# Step 1: server.js 是否存活
pm2 list | grep safemask-dl-auth

# Step 2: 端口监听
ss -tlnp | grep 35347

# Step 3: 直接测试 Node
curl -i http://127.0.0.1:35347/health

# Step 4: 测试 Nginx
curl -i https://www.xiao.org.cn/health

# Step 5: 查看 Nginx 错误日志
tail -30 /var/log/nginx/error.log
```

### 7.2 EADDRINUSE（端口被占用）

**现象**：PM2 日志显示 `code: 'EADDRINUSE', port: 3000` 或 `port: 35347`。

```bash
# 查看占用端口的进程
ss -tlnp | grep 35347

# 清理旧 PM2 进程
pm2 kill
sudo bash /opt/safemask-auth/deploy.sh --restart-only
```

### 7.3 server.js 启动失败

**现象**：`pm2 list` 显示 `errored`，日志无 `✅` 输出。

```bash
# 查看详细错误
pm2 logs safemask-dl-auth --lines 50

# 常见原因：
# - 环境变量 HMAC_KEY 未设置
# - 环境变量 DOWNLOAD_URL 未设置
# - 端口被占用
# - Node.js 版本 < 18（语法不支持）

# 手动启动测试
sudo -E node /opt/safemask-auth/server.js
```

### 7.4 验证全部失败（000）

**原因**：curl 完全连不上目标地址。

```bash
# 检查域名 DNS
nslookup www.xiao.org.cn

# 检查服务器网络
ping www.xiao.org.cn

# 检查防火墙
iptables -L -n | grep 443
```

### 7.5 OSS 文件不可达

**现象**：`verify.sh --download-check` 返回 HTTP 非 200 或 size = 0。

```bash
# 单独测试 OSS 地址
curl -I https://obs.be.com:9004/gxzh/2026/07/06/privacy-filter.zip --max-time 10

# 检查 OSS 是否设置了 IP 白名单 / Referer 防盗链
```

## 8. 升级到 Redis（可选）

当前 `server.js` 使用内存 Map 存储计数，进程重启后计数清零。生产环境建议使用 Redis 持久化。

### 8.1 安装 Redis

```bash
apt install redis-server
systemctl enable --now redis-server
```

### 8.2 修改 server.js

安装依赖：

```bash
cd /opt/safemask-auth
npm init -y
npm install ioredis
```

将 `checkRateLimit` 函数替换为 Redis 版本：

```javascript
const Redis = require('ioredis');
const redis = new Redis();

async function checkRateLimit(sigPart, ip) {
  const tokenKey = `safemask:token:${sigPart}`;
  const tokenCount = await redis.get(tokenKey);
  if (parseInt(tokenCount || '0', 10) >= MAX_PER_TOKEN) return false;
  await redis.incr(tokenKey);
  await redis.expire(tokenKey, TOKEN_TTL_SECS);

  const today = new Date().toISOString().slice(0, 10);
  const ipKey = `safemask:ip:${ip}:${today}`;
  const ipCount = await redis.get(ipKey);
  if (parseInt(ipCount || '0', 10) >= MAX_PER_IP_DAILY) return false;
  await redis.incr(ipKey);
  await redis.expire(ipKey, 86400);

  return true;
}
```

### 8.3 重启服务

```bash
sudo bash /opt/safemask-auth/deploy.sh --restart-only
```

## 9. 监控建议

| 监控项 | 方法 | 告警阈值 |
|--------|------|----------|
| 服务存活 | 外部 HTTP GET `/health`（云监控 / UptimeRobot） | 连续 3 次超时或非 200 |
| SSL 证书到期 | `certbot renew --dry-run` + cron 检查 | 到期前 30 天 |
| 进程异常退出 | PM2 自带重启，无需额外监控 | `pm2 list` 显示 `errored` |
| 磁盘空间 | 监控 `/opt/safemask-auth/config/` 分区 | < 1GB |
| 端口占用 | 定时 `ss -tlnp \| grep 35347` | 无监听 |

推荐添加一个 cron job 定时执行 `verify.sh`：

```bash
# 每 5 分钟通过域名验证一次
*/5 * * * * /opt/safemask-auth/verify.sh --url https://www.xiao.org.cn >/dev/null 2>&1
```

## 11. FAQ

**Q: 服务器重启后服务会自动恢复吗？**

A: 是的。`pm2 startup` + `pm2 save` 配置后，PM2 会在系统重启时自动恢复所有进程。如果未配置，手动执行 `pm2 resurrect` 即可。

**Q: 用户下载流量会不会经过我的服务器？**

A: 不会。鉴权服务器只返回 302 重定向，客户端直接连接 OSS 下载，数据不经过鉴权服务器。

**Q: 如何查看今天的下载统计？**

A: 当前使用内存 Map，无持久化统计。如需日志审计，可开启 PM2 日志并 grep 302 响应行：

```bash
pm2 logs safemask-dl-auth --lines 500 | grep "302"
```

**Q: deploy.sh 执行报错"找不到 node"但 node -v 有输出？**

A: 这是 sudo 的 `secure_path` 隔离问题。deploy.sh 已内置路径扫描（/usr/local/bin、nvm 等），如仍失败请确认 node 在以下路径之一：`/usr/local/bin/node`、`/usr/bin/node`、`/root/.nvm/versions/node/*/bin/node`。

**Q: verify.sh 不需要 HMAC 密钥也能测试前 4 项？**

A: 是的。前 4 项（health / 400 / 404 / 403）不依赖密钥。第 5、6 项需要 `hmac.key` 生成有效 token。`--download-check` 同样需要密钥。
