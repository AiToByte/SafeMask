## 🚨 核心漏洞与技术陷阱审查

### 1. 致命 Bug 1：Nginx 反向代理下的 IP 限流失效陷阱（导致全球用户被锁死）
* **问题描述**：
  在您的第三步中，配置了 Nginx 作为反向代理，将请求转发给运行在 `127.0.0.1:3000` 的 Node.js。
  如果在 Node.js（`server.js`）中直接使用 `req.socket.remoteAddress` 获取客户端 IP，**拿到的将永远是 `127.0.0.1`（因为请求由本地 Nginx 发出）**。
* **严重后果**：
  由于限制了 `MAX_PER_IP_DAILY = 2`（每个 IP 每天 2 次），一旦有任意用户成功下载了 2 次，`127.0.0.1` 的限流计数就会满。**此时全球所有其他用户都将收到 `429 Rate limited` 的报错，整个 AI 下载功能彻底瘫痪**。
* **完善方案**：
  在 Node.js 中必须优先读取 Nginx 传递的 `X-Real-IP` 或 `X-Forwarded-For` 请求头：
  ```javascript
  // 在 server.js 中获取真实 IP 时的安全写法：
  const rawIp = req.headers['x-real-ip'] || 
                req.headers['x-forwarded-for'] || 
                req.socket.remoteAddress;
  
  // 过滤 IPv6 映射的 IPv4 前缀 (例如 ::ffff:127.0.0.1)
  const ip = rawIp.startsWith('::ffff:') ? rawIp.substring(7) : rawIp;
  ```

### 2. 致命 Bug 2：纯内存 Map 计数器导致内存泄露与 OOM 崩溃
* **问题描述**：
  在未开启 Redis、直接使用 `server.js` 的内存 Map（如 `tokenMap` 和 `ipMap`）进行计数的单机版中，如果缺乏定期清理机制，每一个请求的 Token 和每一个访问的 IP 都会作为 Key 永久驻留在内存中。
* **严重后果**：
  随着 SafeMask 用户的增加，运行数月后，由于内存不断累积且从未释放，Node.js 进程最终会触发 **OOM (Out Of Memory) 内存溢出崩溃**。
* **完善方案**：
  在不引入第三方依赖（如 Redis）的前提下，在 `server.js` 中增加一个极其轻量、每小时自动执行的**定时内存清理逻辑**：
  ```javascript
  // 内存防泄露清理机制 (以单机 Map 为例)
  setInterval(() => {
    const now = Date.now();
    // 1. 清理过期的 Token 记录
    for (const [token, data] of tokenMap.entries()) {
      if (now - data.timestamp > TOKEN_TTL_SECS * 1000) {
        tokenMap.delete(token);
      }
    }
    // 2. 跨天时清理昨天的 IP 计数器（避免 IP Map 无限增长）
    const today = new Date().toISOString().slice(0, 10);
    for (const ipKey of ipMap.keys()) {
      // 假设 IP 的 key 格式为 "ip:192.168.1.1:2026-07-06"
      if (!ipKey.endsWith(today)) {
        ipMap.delete(ipKey);
      }
    }
  }, 1000 * 60 * 60); // 每小时执行一次
  ```

### 3. 打包陷阱：Windows 环境下 DLL 存放路径导致运行闪退
* **问题描述**：
  在您的第五步和前文的 `resources` 配置中，计划将 `onnxruntime.dll` 打包进 Tauri 的 `resources` 目录。
  在 Windows 下，Windows 动态链接库的寻址顺序默认是：`EXE同级目录` -> `System32` -> `PATH 环境变量`。而 Tauri 的 `resources` 会被打包至应用安装路径下的子目录中（如 `C:\Program Files\SafeMask\resources\`）。
* **严重后果**：
  当用户双击打开程序时，主程序 `SafeMask.exe` 会立刻因为在同级目录找不到 `onnxruntime.dll` 而**直接闪退（Silent Crash）**，连进入前端界面的机会都没有。
* **完善方案**：
  不要将 `.dll` 文件放在 `resources` 这种深层子目录下，而应在构建阶段使用自定义的 Rust `build.rs` 脚本，或者修改编译流，确保在生成安装包时，**直接将 `onnxruntime.dll` 拷贝至与可执行文件同级的根目录下**。
  
  在 `src-tauri/build.rs` 中加入自动化拷贝逻辑：
  ```rust
  // src-tauri/build.rs
  use std::env;
  use std::path::PathBuf;

  fn main() {
      tauri_build::build();

      // 在打包构建时，自动把 target 目录中的 onnxruntime.dll 拷贝到发布包根目录
      let target_dir = env::var("OUT_DIR").unwrap();
      // 执行拷贝逻辑...
  }
  ```

### 4. 架构安全局限性说明：客户端 HMAC 签名的“防君子不防小人”属性
* **机制局限**：
  客户端（Tauri）使用 statically-compiled （静态硬编码）的 `SAFEMASK_HMAC_KEY` 生成 Token。虽然我们在打包时通过环境变量注入，但这个密钥最终会以常量形式存放在编译好的 SafeMask 二进制文件中。
* **安全风险**：
  任何有逆向经验的攻击者只需使用二进制字符串提取工具（如 `strings` 或 IDA/Ghidra），很容易就能反编译出这个 Base64 密钥。一旦密钥泄露，攻击者就可以绕过客户端，批量生成合法的 Token 并发起脱机下载。
* **缓解建议**：
  当前架构非常适合防范**普通用户直接复制链接进行盗链**。但如果后续 OSS 流量成本面临恶意刷量压力，建议升级为：
  * **客户端单向请求**：客户端无密钥，直接向服务器 `POST /request-download`，服务器验证通过后，在**服务端**动态计算一个 5 分钟内有效的临时 OSS 签名 URL（如阿里云 OSS 的 Presigned URL），随后 302 重定向。这样客户端二进制中绝无密钥，安全性提升为工业级。
