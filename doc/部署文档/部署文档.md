# SafeMask AI 模型下载鉴权代理 — 部署文档

## 1. 架构概览

```
┌──────────────────┐ ① GET /download?token=<令牌>  ┌──────────────────────┐
│  SafeMask 客户端  │ ───────────────────────────→   │   自建鉴权服务器      │
│  (Rust + Tauri)   │                                │  (Node.js + Nginx)   │
│                   │←─── ② 302 Redirect ──────────│                      │
│                   │                                │  1. HMAC-SHA256 验证  │
│   ③ 跟随 302     │                                │  2. TTL 检查 (1 小时) │
│   直连 OSS        │ ───────────────────────────→   │  3. 设备指纹比对       │
│                   │        OSS / R2 / GitHub       │  4. 频率限制           │
└──────────────────┘                                └──────────────────────┘
```

**核心设计**：服务端只做 **302 重定向，不代理数据流**。验证通过后客户端直连 OSS 下载，鉴权服务器带宽成本为零。

**组件端口映射**：

| 组件 | 端口 | 说明 |
|------|------|------|
| server.js（Node.js） | 35347 | 鉴权服务监听端口 |
| Nginx | 443 (HTTPS) | 反向代理 35347，终结 SSL |
| PM2 | — | 守护 server.js 进程 |

## 3. 文件清单

从项目仓库上传到服务器的文件：

| 文件 | 大小 | 作用 |
|------|------|------|
| `server.js` | ~5 KB | 鉴权服务（零依赖，仅 Node.js 内置 `http` + `crypto`） |

| `deploy.sh` | ~12 KB | 一键部署脚本（密钥 → PM2 → 验证） |
| `nginx.conf.template` | ~1 KB | Nginx 配置参考模板 |
| `verify.sh` | ~5 KB | 部署验证脚本（独立，无依赖） |

## 4. 服务器部署步骤

### 4.1 上传文件

将 5 个文件上传到服务器同一目录：

```bash
scp workers/download-proxy/server.js            root@你的IP:/opt/safemask-auth/
scp workers/download-proxy/init-key.sh          root@你的IP:/opt/safemask-auth/
scp workers/download-proxy/deploy.sh            root@你的IP:/opt/safemask-auth/
scp workers/download-proxy/nginx.conf.template  root@你的IP:/opt/safemask-auth/
scp workers/download-proxy/verify.sh            root@你的IP:/opt/safemask-auth/
```

设为可执行：

```bash
chmod +x /opt/safemask-auth/*.sh
```

> 所有文件必须位于**同一目录**（如 `/opt/safemask-auth/`），因为 `deploy.sh` 和 `verify.sh` 依赖同目录下的脚本文件。

### 4.2 生成 HMAC 密钥

```bash
sudo bash /opt/safemask-auth/init-key.sh
```

- 密钥保存在 `/opt/safemask-auth/config/hmac.key`（chmod 600）
- 环境变量文件 `/opt/safemask-auth/config/safemask.env` 同步生成
- 幂等：再次运行不覆盖已有密钥（除非 `--force`）

> 该密钥同时用于服务端（server.js 验证令牌）和客户端（Rust 二进制生成令牌）。**务必保存一份副本**用于后续 Rust 构建。

### 4.3 一键部署

```bash
cd /opt/safemask-auth
sudo bash deploy.sh \
  --domain www.xiao.org.cn \
  --download-url https://obs.be.com:9004/gxzh/2026/07/06/privacy-filter.zip
```

`deploy.sh` 自动执行：

```
① 前置检查 → ② HMAC 密钥（调用 init-key.sh） → ③ 部署 server.js → ④ PM2 启动 → ⑤ 6 项验证
```

参数说明：

| 参数 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `--domain` | 是 | — | 鉴权服务域名 |
| `--download-url` | 是 | — | OSS 真实下载地址 |
| `--port` | 否 | 35347 | Node.js 监听端口 |
| `--key-dir` | 否 | `/opt/safemask-auth/config` | 密钥存储目录 |
| `--config-dir` | 否 | `/opt/safemask-auth` | server.js 所在目录 |
| `--dry-run` | 否 | false | 仅打印操作，不执行 |
| `--restart-only` | 否 | false | 仅重启 PM2，跳过验证 |
| `--skip-verify` | 否 | false | 跳过部署后验证 |

### 4.4 配置 Nginx

> 手动执行，仅首次部署一次。

参考 `nginx.conf.template`，写入 `/etc/nginx/sites-available/safemask-auth`：

```nginx
server {
    listen 443 ssl http2;
    server_name www.xiao.org.cn;

    ssl_certificate     /etc/letsencrypt/live/www.xiao.org.cn/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/www.xiao.org.cn/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        proxy_pass http://127.0.0.1:35347;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

server {
    listen 80;
    server_name www.xiao.org.cn;
    return 301 https://$server_name$request_uri;
}
```

启用并重载：

```bash
ln -sf /etc/nginx/sites-available/safemask-auth /etc/nginx/sites-enabled/
nginx -t
nginx -s reload
```

#### SSL 证书获取（如未配置）

```bash
apt install certbot python3-certbot-nginx
certbot --nginx -d www.xiao.org.cn
certbot renew --dry-run
```

#### Nginx 层频率限制（可选）

在 `/etc/nginx/nginx.conf` 的 `http {}` 块中添加：

```nginx
limit_req_zone $binary_remote_addr zone=safemask_dl:5m rate=1r/s;
limit_conn_zone $binary_remote_addr zone=conndl:5m;
```

然后在 location 中取消注释对应行。建议先观察正常流量基线再启用。

### 4.5 验证部署

```bash
bash /opt/safemask-auth/verify.sh                           # 6 项基础验证
bash /opt/safemask-auth/verify.sh --url https://www.xiao.org.cn  # 通过 Nginx 测试
bash /opt/safemask-auth/verify.sh --download-check          # + OSS 文件可达性
```

验证项说明：

| # | 端点 | 期望 HTTP | 覆盖功能 |
|---|------|-----------|----------|
| 1 | `GET /health` | 200 | 服务存活 + 健康检查端点 |
| 2 | `GET /download`（无 token） | 400 | 缺少参数的拒绝 |
| 3 | `GET /`（错误路径） | 404 | 错误路由的拒绝 |
| 4 | `GET /download?token=invalid` | 403 | HMAC 签名验证 |
| 5 | `GET /download?token=<有效>` | 302 | 完整鉴权 + 重定向 |
| 6 | 同一 token 再请求 | 429 | 频率限制 |

## 5. 客户端配置

### 5.1 修改 Worker 域名

编辑 `src-tauri/src/core/download_auth.rs`：

```rust
// 修改前（占位符）
pub const WORKER_BASE_URL: &str = "https://safemask-dl-proxy.xxx.workers.dev";

// 修改后（你的服务域名，不带尾部斜杠）
pub const WORKER_BASE_URL: &str = "https://www.xiao.org.cn";
```

### 5.2 构建 Rust 二进制

```bash
# 读取服务器上的 HMAC 密钥
ssh root@你的服务器 "cat /opt/safemask-auth/config/hmac.key"

# Windows PowerShell
$env:SAFEMASK_HMAC_KEY = "上一步获取的密钥"
cargo build -p SafeMask --release

# Linux / macOS
SAFEMASK_HMAC_KEY="上一步获取的密钥" cargo build -p SafeMask --release
```

> 开发环境不设置 `SAFEMASK_HMAC_KEY` 时自动使用 `dev-fallback-key`（与服务端 `dev-fallback-key` 匹配），安全但建议生产构建时务必设置。

## 6. 环境变量参考

| 变量 | 设置位置 | 作用 | 默认值 |
|------|----------|------|--------|
| `HMAC_KEY` | server.js 环境变量 | HMAC 签名密钥 | `dev-fallback-key` |
| `DOWNLOAD_URL` | server.js 环境变量 | OSS 真实下载地址 | 无（必填） |
| `PORT` | server.js 环境变量 | 监听端口 | 3000 |
| `SAFEMASK_HMAC_KEY` | Rust 构建时环境变量 | 与 `HMAC_KEY` 一致 | `dev-fallback-key` |
| `TOKEN_TTL_SECS` | server.js 常量 | 令牌有效期（秒） | 3600 |
| `MAX_PER_TOKEN` | server.js 常量 | 每令牌最大下载次数 | 1 |
| `MAX_PER_IP_DAILY` | server.js 常量 | 每 IP 每天最大下载次数 | 2 |
