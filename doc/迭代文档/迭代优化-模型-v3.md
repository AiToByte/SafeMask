通过对您提供的原文和非预期脱敏结果的深度比对，我们精准定位到了当前系统在混合引擎协作时的**底层核心病因**。

这是一个非常隐蔽且典型的**“二次换算判定陷阱”**。导致此现象的根本原因以及解决方案分析如下：

---

### 一、 核心病因诊断

#### 1. 致命的判定失效：`verify_byte_offsets` 的“假阴性”
在之前的优化中，为了兼容不同类型的 Tokenizer 配置，代码引入了 `verify_byte_offsets` 校验函数。然而在实际运行中，该函数存在一个技术漏洞：
* **单 Token 解码不匹配**：该函数通过对单个 Token ID 调用 `tokenizer.decode()` 并与原文中的字节切片进行比对。但是在 Byte-Level BPE 分词器中，许多子词或空格标志（如 `Ġ`）在独立解码时无法与原文的 UTF-8 字节完全对齐，或者在多字节字符截断时会被判断为不匹配。
* **判定结果归零**：由于校验函数中只要有一个 Token 不等就立即返回 `false`，导致 `verify_byte_offsets` 在运行中文或复杂文本时**无条件返回 `false`**。

#### 2. 被扭曲的二次重投影（Offsets Double Conversion）
* **病因推导**：
  在 HuggingFace 的 Rust 原生 `tokenizers` 库中，调用 `encoding.get_offsets()` 返回的**本来就是 100% 正确的 UTF-8 字节偏移量（Byte Offsets）**，这与 Rust 原生字符串切片 `&str[start..end]` 的字节指针需求完全一致。
  但由于 `verify_byte_offsets` 误判返回了 `false`，引擎走入了 `else` 分支，强行对已经是字节偏置的 `raw_offsets` 数据执行了 `char_to_byte` 的重映射换算。
  这使得原本正确的字节偏移量被大幅向右推移（例如把实际的字节 118 换算成了字符 118 对应的更大字节值）。

#### 3. 乱码与覆盖异常分析
由于 AI 引擎输出的偏移量被整体向右错位：
* **`[EMAIL]<EMAIL>` 并排出现**：AI 识别的邮箱由于偏移量右移，在物理位置上与正则识别的 `<EMAIL>` **完全错开**。冲突解决器（`ConflictResolver`）判定两者不重叠，因而让它们并排输出，没有触发优先级消解。
* **`https[URL]angsan` 破裂**：URL 的脱敏区间整体向右发生了字节级漂移，破坏了原本的 `://`，并在后面留下了 `angsan` 破裂明文。
* **尾部碎屑堆积**：所有的 AI 标签跨度溢出到了文本的最末尾（`ZhangSan_2024`），导致最后的 QQ 号和微信号被错位的 `[BANK_CARD]`、`[PERSON]` 等标签彻底绞碎。

---

### 二、 解决方案

1. **直接移除多余校验与换算**：
   直接、无条件地信任并提取 `encoding.get_offsets()` 的输出。
2. **剔除冗余代码**：
   在 `src-tauri/src/infra/ai/ner_engine.rs` 中彻底删除 `verify_byte_offsets` 和 `build_char_to_byte_map` 函数。这不仅能修复对齐，还能免去每次推理时的多余字符串遍历，提升推理效率。

---

### 三、 核心重构代码

请将 `src-tauri/src/infra/ai/ner_engine.rs` 文件中 `infer` 函数及相关校验辅助函数的实现重构如下：

```rust
// src-tauri/src/infra/ai/ner_engine.rs

impl NerEngine {
    /// 执行 NER 推理（已移除多余的 verify_byte_offsets 与 char_to_byte 映射）
    pub fn infer(&mut self, text: &str) -> Result<Vec<EntitySpan>> {
        if text.trim().is_empty() {
            return Ok(Vec::new());
        }

        // 1. Tokenize (Rust tokenizers 默认输出即为完美的 UTF-8 字节偏移量)
        let encoding = self.tokenizer.encode(text, false)
            .map_err(|e| anyhow::anyhow!("Tokenize 失败: {}", e))?;

        let token_ids: Vec<u32> = encoding.get_ids().to_vec();
        let attention_mask: Vec<u32> = encoding.get_attention_mask().to_vec();
        
        // 🚀 核心修正：直接、无条件采用原生的字节偏移
        let offsets: Vec<(usize, usize)> = encoding.get_offsets().to_vec();
        let seq_len = token_ids.len().min(self.max_length);

        let input_ids_i64: Vec<i64> = token_ids[..seq_len].iter().map(|&x| x as i64).collect();
        let attention_mask_i64: Vec<i64> = attention_mask[..seq_len].iter().map(|&x| x as i64).collect();

        // 2. 创建输入张量
        let input_ids_tensor = Tensor::from_array(([1, seq_len], input_ids_i64))?;
        let attention_mask_tensor = Tensor::from_array(([1, seq_len], attention_mask_i64))?;

        // 3. 执行推理并提取 logits
        let (seq_len_actual, num_labels, logits_owned) = {
            let outputs = self.session.run(ort::inputs![
                "input_ids" => input_ids_tensor,
                "attention_mask" => attention_mask_tensor,
            ])
            .map_err(|e| anyhow::anyhow!("ONNX 推理失败: {}", e))?;

            let logits_value = outputs.get("logits").context("模型输出中未找到 logits")?;
            let (shape, logits_slice) = logits_value.try_extract_tensor::<f32>()?;

            let seq = shape[1] as usize;
            let labels = shape[2] as usize;
            let data: Vec<f32> = logits_slice.to_vec();

            (seq, labels, data)
        };

        // 4. 重建 ndarray 视图
        let logits_view = ndarray::ArrayView3::from_shape(
            (1, seq_len_actual, num_labels),
            &logits_owned,
        ).context("重建 logits 张量失败")?;

        // 5. BIOES 后处理
        let spans = self.decode_bioes(&logits_view, &offsets, text, seq_len_actual);

        Ok(spans)
    }
}

```
