# 待确认 & 实施计划 — AI 模型按需下载模块

## 📌 待确认项

### Q1: 模型发布形式 — 单归档 vs 多文件直链

`src-tauri/models/privacy-filter/` 当前文件清单：

| 文件 | 大小 | 用途 |
|------|------|------|
| `model_q4.onnx` | ~160 KB | ONNX 模型结构（含内联 metadata） |
| `model_q4.onnx_data` | ~917 MB | ONNX 外部权重（运行时必须随行） |
| `tokenizer.json` | ~28 MB | HuggingFace 分词器 |
| `config.json` | ~3 KB | 标签映射（id2label） |
| `.cache/` | ~900 MB | HuggingFace 下载缓存（不应分发） |

**需要确认**：GitHub Release 上以什么形式提供？

- **A) 单 zip 包**：`privacy-filter.zip`（不含 `.cache/`），Rust 端需加 `zip` crate 解压到 `app_local_data_dir`
- **B) 逐个文件直链**：串行下载 `model_q4.onnx` → `model_q4.onnx_data` → `tokenizer.json` → `config.json`
- **C) 仅托管 `model_q4.onnx` + `model_q4.onnx_data`，tokenizer 和 config 随安装包分发**（小文件）

### Q2: 下载 URL 与 Release tag

v1.1 文档使用 `https://github.com/AiToByte/SafeMask/releases/download/v1.2.4/model.onnx`。

**需要确认**：
- 新 Release 的 tag 名称（保持 `v1.2.4` 还是开新 tag？）

- 还是使用其他免费的或者OSS 存储服务, CND 存储等

### Q3: 是否保留便携版手动放置路径

目前 `find_models_dir()` 会检查 `resource_dir() / models`。若用户手动将模型文件夹放到安装目录的 resources 下（便携场景），是否仍需优先识别？

- 若需要 → 保留 `resource_dir` 优先查找逻辑
- 若不需要 → 统一走 `app_local_data_dir`，简化路径逻辑

### Q4: `ort` crate 的 `copy-dylibs` feature

**需要确认**：`src-tauri/Cargo.toml` 中 `ort` 依赖当前是否已启用 `copy-dylibs`。

### Q5: 安装后 ONNX Runtime DLL 缺失

v1 文档指出 ONNX Runtime 动态库（`onnxruntime.dll`）可能在打包后丢失。是否已有此问题复现？

## 📋 实施计划（三个阶段）

### 阶段一：Rust 下载内核

#### 1.1 新建 `src-tauri/src/ai_downloader.rs`

基于 v1.1 文档代码，修正点：

- 将 v1.1 中所有 `model.onnx` 替换为 `model_q4.onnx`
- `check_model_file` 校验四文件完整性（`model_q4.onnx` + `model_q4.onnx_data` + `tokenizer.json` + `config.json`），而非单文件
- 若 Q1 选 A（zip 包），增加 `zip` crate 解压逻辑替代逐文件写入
- 若 Q1 选 B（多文件），为每个文件生成独立进度还是合并进度？
- 下载目录：`app.path().app_local_data_dir() / "models" / "privacy-filter"`

核心函数：

```rust
#[tauri::command]
pub fn check_model_file(app: AppHandle) -> Result<String, String>

#[tauri::command]
pub async fn start_model_download(
    app: AppHandle,
    state: tauri::State<'_, DownloadState>,
    url: String,
) -> Result<String, String>

#[tauri::command]
pub fn cancel_model_download(state: tauri::State<'_, DownloadState>) -> Result<(), String>
```

#### 1.2 修改 `src-tauri/Cargo.toml`

添加依赖（按 Q1 的答案决定是否需要 `zip` crate）：

```toml
reqwest = { version = "0.12", features = ["stream"] }
futures-util = "0.3"
# zip = "2"  # 如果选 zip 包分发
```

确认 `ort` 已启用 `copy-dylibs` 特性。

#### 1.3 修改 `src-tauri/src/main.rs`

操作：
1. 文件头：`mod ai_downloader;`
2. `Builder::default()` 链中添加 `.manage(ai_downloader::DownloadState::default())`
3. `invoke_handler(generate_handler![...])` 中注册三个新 command
4. 重写 `find_models_dir()` — 替换当前 6 路径猜谜：

```
find_models_dir(handle):
  1. resource_dir() / models /             → 存在且含模型 ⇒ 返回（便携场景）
  2. app_local_data_dir() / models /        → 存在且含模型 ⇒ 返回（下载部署场景）
  3. 都不存在 ⇒ 写入触发信号到 AppState     → 前端 bootstrap 检查后展示下载卡
  4. 返回 app_local_data_dir() / models /   → 作为目标路径（尚不存在，下载后自动创建）
```

### 阶段二：前端状态 & UI

#### 2.1 新建 `src/hooks/useModelDownloader.ts`

状态机：

```
IDLE ──checkStatus()──→ CHECKING
CHECKING ──检测结果──→ MISSING | READY | CORRUPTED
MISSING ──startDownload()──→ DOWNLOADING
DOWNLOADING ──完成──→ READY
DOWNLOADING ──失败──→ ERROR
DOWNLOADING ──取消──→ MISSING
ERROR ──startDownload()──→ DOWNLOADING
```

监听 Rust 端两个 IPC 事件：
- `model-download-progress`（限流 100ms）：百分比、速度、已下载、总量
- `model-download-status`（终态信号）：`READY` / `ERROR:...`

#### 2.2 新建 `src/components/settings/ModelDownloadCard.tsx`

遵循 Amber Ivory 暗黑美学，与现有 Settings 卡片风格一致：

- **READY**：绿色呼吸灯 + "已装载" 徽章 + 确认提示
- **MISSING / CORRUPTED**：琥珀色警告 + "初始化下载" 按钮
- **DOWNLOADING**：磨砂进度条 + 速度/ETA 仪表 + 取消按钮
- **ERROR**：红色错误详情 + "重新连接" 按钮

#### 2.3 嵌入到 `SettingsPage.tsx`

找到 AI 引擎区域（`aiEngineStatus` 所在的 Diagnostics 卡片附近），在合适位置渲染 `<ModelDownloadCard />`。

## ⚠️ 风险项

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 917MB 下载中途断网 | 用户感知差 | 原子写入 + 断点续传（需在后续迭代增加 HTTP Range） |
| ONNX Runtime DLL 未随包分发 | AI 模块闪退 | 确认 `ort` 的 `copy-dylibs` 已启用；CI 构建后核验 target 目录 |
| 模型文件更新（新版本） | 旧文件残留 | 下载前比对版本号 / checksum，自动清理旧文件 |
| 用户磁盘空间不足 | 下载失败 | 下载前校验可用空间（`fs2` crate） |
