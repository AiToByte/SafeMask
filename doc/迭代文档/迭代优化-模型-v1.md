将 SafeMask 程序打包（`tauri build`）并重新安装后，AI 模型等文件无法识别是一个非常典型且多发的 **“打包资源丢失与路径漂移”** 问题。

---

### 🔍 核心原因分析

在开发模式下（`tauri dev`），程序的运行工作目录（CWD）是您的项目根目录，因此在 Rust 中使用像 `./models/privacy-filter/model.onnx` 这样的相对路径可以正常工作。

但在执行 `tauri build` 打包安装后，由于以下三个原因，模型会彻底失效：

1. **资源未打包进安装包（Resources 未声明）**：
   默认情况下，Tauri **只会**打包前端静态编译产物（HTML/CSS/JS）以及 Rust 的可执行二进制文件。任何自定义的外部文件夹（如存放 AI 模型的 `models/`）若不显式配置，**在打包时会被全部忽略**，安装包里根本没有这些文件。
2. **物理安装路径漂移**：
   安装后，程序的安装位置变成了系统限制的物理路径（如 Windows 的 `C:\Program Files\SafeMask\` 或 macOS 的 `/Applications/SafeMask.app/`）。由于工作目录发生变化，代码中直接使用的 `./models` 相对路径会指向错误的操作系统上下文，导致“文件未找到”的错误。
3. **ONNX Runtime 动态链接库丢失（DLL/dylib Missing）**：
   如果您的 Rust AI 引擎使用了 `ort` (ONNX Runtime) 库，它在运行时依赖 `onnxruntime.dll` (Windows) 或 `libonnxruntime.dylib` (macOS)。这些动态链接库虽然在开发时存在于本地 `target/` 中，但如果不进行手动资源搬运配置，打包出来的安装程序运行时会因为**找不到 ONNX 动态库**而直接导致 AI 模块崩溃崩溃或闪退。

---

### 🛠️ 解决方案

#### 第一步：在 `tauri.conf.json` 中配置资源打包规则
我们需要在 Tauri 配置文件中将模型文件声明为**打包资源（Resources）**。
打开 `src-tauri/tauri.conf.json`，在 `bundle` 节点下配置 `resources`：

```json
{
  "bundle": {
    "resources": [
      "../models/**/*"
    ]
  }
}
```
> *注：`../models/**/*` 表示将 `src-tauri` 目录同级的 `models` 文件夹及其内部所有子文件在打包时一同拷贝至安装包的特定资源目录下。*

---

#### 第二步：在 Rust 中使用 Tauri 路径解析器（Path Resolver）
在 Rust 代码中，**绝对不要使用手动拼接的相对路径**，必须使用 **Tauri v2** 的动态路径解析器获取资源在当前操作系统下的绝对路径。

修改 Rust 中加载模型文件的代码：

```rust
use tauri::Manager;
use tauri::path::BaseDirectory;

// 假设这是您初始化 AI 模型的 Rust Command 函数
#[tauri::command]
pub fn load_ai_model(app_handle: tauri::AppHandle) -> Result<(), String> {
    // 🚀 使用 Tauri v2 路径解析器动态定位资源路径
    let model_path = app_handle
        .path()
        .resolve("models/privacy-filter/model.onnx", BaseDirectory::Resource)
        .map_err(|e| format!("无法定位 AI 模型资源路径: {}", e))?;

    // 此时的 model_path 在安装包中是绝对安全的绝对路径，例如：
    // C:\Program Files\SafeMask\resources\models\privacy-filter\model.onnx
    println!("成功定位模型物理路径: {:?}", model_path);

    // 接下来将 model_path 传入 ort / ONNX 引擎中加载即可
    // your_onnx_loader.load(model_path);

    Ok(())
}
```

---

#### 第三步：解决 `onnxruntime` 动态链接库丢失
如果安装后，程序在加载 AI 时无任何报错直接闪退，通常是缺少 ONNX Runtime 运行库。

1. 在 `src-tauri/Cargo.toml` 中，为 `ort` 依赖项启用 `copy-dylibs` 特性，这会确保编译时库文件会自动拷贝到目标路径：
   ```toml
   [dependencies]
   ort = { version = "2.0.0", features = ["copy-dylibs"] }
   ```
2. 同时在 `tauri.conf.json` 的 `resources` 中，将构建生成的动态链接库也作为资源打包进去（防止目标机器没有安装 ONNX 环境）：
   ```json
   {
     "bundle": {
       "resources": [
         "../models/**/*",
         "target/release/*.dll", // Windows
         "target/release/*.dylib" // macOS
       ]
     }
   }
   ```

### 💡 最佳生产实践：大模型文件的动态按需下载

因为您的 AI 模型文件有 **874MB**，直接打包进 `.msi` 或 `.dmg` 安装包会导致：
* 安装包极其庞大（用户下载困难）；
* 极易触发 Tauri/WiX 编译器对单个文件体积过大或内存溢出的构建报错（如打包工具崩溃）。

**现代化桌面 AI 应用（如 Ollama、LM Studio）的通用做法**：
1. **轻量化安装包**：安装包本身只有十几兆，不包含 874MB 的模型。
2. **本地数据目录存储**：将模型文件存放在用户的 Local AppData（应用专属数据目录）中。
3. **按需下载**：当用户在设置页面首次打开 "AI NER 识别引擎" 时，程序检测本地是否存在该文件，若不存在，则在前端显示下载进度条，在 Rust 端将其从 CDN（如 GitHub Release / 阿里云 OSS / S3）中下载到本地。

**Rust 代码实现逻辑**：
```rust
use tauri::Manager;

#[tauri::command]
pub fn check_and_get_model_path(app: tauri::AppHandle) -> Result<std::path::PathBuf, String> {
    // 获取用户本机的 Local AppData 目录：
    // Windows: C:\Users\用户名\AppData\Local\SafeMask\
    // macOS: /Users/用户名/Library/Application Support/SafeMask/
    let app_data_dir = app.path()
        .app_local_data_dir()
        .map_err(|e| e.to_string())?;

    let model_dir = app_data_dir.join("models").join("privacy-filter");
    let model_file = model_dir.join("model.onnx");

    if !model_file.exists() {
        // 返回特定信号，提示前端调用下载 API 
        return Err("MODEL_NOT_FOUND".to_string());
    }

    // 若存在，直接返回物理路径供引擎加载
    Ok(model_file)
}
```
这种设计会让您的程序构建过程非常丝滑，同时用户的安装和更新体验也会成倍提升。
