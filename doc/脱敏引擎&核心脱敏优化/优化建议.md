### 一、 核心计划审计与缺陷修正

#### 1. 潜在缺陷 1：`mask_email` 中的 UTF-8 字符切片崩溃隐患 (Phase 4.2)
* **计划中的写法**：
  ```rust
  let prefix = &username[..visible]; // 🚀 潜在崩溃点
  ```
* **技术病因**：
  在 Rust 中，对 `&str` 进行直接切片（如 `username[..visible]`）使用的是**字节索引（Byte Index）**，而非字符索引。
  如果用户在邮箱的用户名中使用了多字节的中文或特殊字符（例如：`张三@company.cn`），由于一个中文字符在 UTF-8 下占 3 个字节，当 `visible` 计算为 2 并试图切片 `username[..2]` 时，会因为**切在了非字符边界上（Not a char boundary）直接触发系统级崩溃（Panic）**。
* **高质量修正方案**：
  使用 `chars()` 收集器来安全处理前缀提取，彻底杜绝多字节字符带来的越界和切分崩溃：
  ```rust
  fn mask_email(original: &str) -> String {
      if let Some(at_idx) = original.find('@') {
          let username = &original[..at_idx];
          let domain = &original[at_idx..];
          let chars: Vec<char> = username.chars().collect();
          let char_len = chars.len();
          let visible = match char_len {
              0..=1 => 0,
              2..=4 => 1,
              _ => 2.min(char_len.saturating_sub(1)),
          };
          let prefix: String = chars[..visible].iter().collect();
          let mask_len = char_len.saturating_sub(visible).max(1);
          format!("{}{}{}", prefix, "*".repeat(mask_len), domain)
      } else {
          Self::mask_fallback(original)
      }
  }
  ```

#### 2. 潜在缺陷 2：`mask_phone` 格式吞噬问题 (Phase 4.3)
* **计划中的写法**：
  通过 `original.chars().filter(|c| c.is_ascii_digit()).collect()` 提取出纯数字进行局部遮盖。
* **技术病因**：
  这样处理虽然简单，但会**完全吞噬原文本中的非数字字符（格式符）**。
  例如，若输入为带格式的电话 `"+86 138-0013-8000"`，脱敏后将变成纯数字 `"861******8000"`，丢失了原有的国别前缀 `+`、空格和 `-` 连字符，严重降低了高保真还原的体验。
* **高质量修正方案**：
  采用 **“原样格式保留的动态数字遮盖算法”**（Lazy Digit Masking）。只对原字符串中的数字字符进行索引计数和遮盖，非数字字符（如 `+`、`-`、空格）直接原样放行，保留其物理位置：
  ```rust
  fn mask_phone(original: &str) -> String {
      let total_digits = original.chars().filter(|c| c.is_ascii_digit()).count();
      if total_digits < 7 {
          return Self::mask_fallback(original);
      }

      let (mask_start, mask_end) = if total_digits >= 11 {
          (3, total_digits - 4) // 138-0013-8000 -> 保持前3位与后4位数字不遮盖
      } else {
          (2, total_digits - 2)
      };

      let mut digit_idx = 0;
      original.chars().map(|c| {
          if c.is_ascii_digit() {
              let current = digit_idx;
              digit_idx += 1;
              if current >= mask_start && current < mask_end {
                  '*'
              } else {
                  c
              }
          } else {
              c
          }
      }).collect()
  }
  ```
  * **效果**：`"+86 138-0013-8000"` $\rightarrow$ `"+86 138-****-8000"`。完美保留结构，仅脱敏核心标识。

### 二、 针对 `is_useless_fragment` 算法的高质量升级建议 (Phase 2.1)

在 Phase 2.1 规划的 `is_useless_fragment` 辅助函数中，只对 ASCII 标点符号进行了检测：
```rust
if fragment.len() == 1 && fragment[0].is_ascii_punctuation() { return true; }
```
由于 SafeMask 在处理中文日志和住址时会遇到大量**全角标点符号**（如 `，`、`。`、`：` 等多字节 Unicode 字符），单字节的 `is_ascii_punctuation()` 判定会彻底失效。

#### 建议升级为：中英文兼容的 UTF-8 语义碎片剪裁算法

```rust
fn is_useless_fragment(span: &EntitySpan, text: &[u8]) -> bool {
    if span.start >= span.end || span.end > text.len() { return true; }
    let fragment_bytes = &text[span.start..span.end];
    
    if let Ok(s) = std::str::from_utf8(fragment_bytes) {
        let trimmed = s.trim();
        if trimmed.is_empty() { return true; }
        
        // 1. 如果碎片仅包含标点符号（兼容中英文全角半角标点）
        if trimmed.chars().all(|c| {
            c.is_ascii_punctuation() 
            || "，。：；！？、‘’“”【】（）—《》".contains(c)
        }) {
            return true;
        }
        
        // 2. 针对复杂高保真语义实体（住址、姓名），如果雕刻后留下的碎屑长度仅剩 1 个字，
        // 例如地址被拦截后只剩下孤立的 "号"、"室"、"区"、"市"、"省"，则视为无价值噪点直接剪裁
        let char_count = trimmed.chars().count();
        if char_count < 2 {
            if matches!(span.entity_type, EntityType::Address | EntityType::Person | EntityType::BankCard) {
                return true;
            }
        }
    } else {
        // 非合法 UTF-8 字节序列（可能是断层切片），直接剪裁防错
        return true;
    }
    
    false
}
```
